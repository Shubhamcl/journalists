# Image Policy

## Goal

Use images to improve clarity, trust, and click-through rate. Do not use images as filler.

## Default image mix

Prefer this order:

1. original screenshots
2. original photos taken by us when available
3. official product, company, or event press images where editorial reuse is clearly allowed
4. Wikimedia Commons or other openly licensed images with per-file license verification
5. Openverse-discovered images only after verifying the source license on the underlying source page
6. generic free stock from sources like Unsplash or Pexels only for non-specific contextual use

Do not use AI-generated images for now.

Avoid generic stock photos unless there is no practical alternative.

## By article type

### How-to

- annotated screenshots
- step visuals
- before/after UI examples

### Comparison or roundup

- comparison table graphic
- product or UI screenshots
- product photos or official product renders when rights are clear
- feature callout graphics

### Explainer

- diagrams
- process visuals
- simple infographics
- licensed editorial photos when the story is about a real company, person, event, or device

### Life or hybrid posts

- real workspace or device context when relevant
- restrained editorial hero images
- visuals that still suggest technology, not generic lifestyle filler

## Hero image rules

The hero image should:

- look specific to the article topic
- support social sharing previews
- avoid misleading claims
- match the tone of the site

Do not use the same visual template so aggressively that every article looks identical.

When the article is about a specific real product or announcement, prefer a real image of that product rather than an abstract concept image.

## SEO and performance rules

For every image:

- use descriptive filenames
- write useful alt text
- prefer WebP or AVIF when practical
- keep dimensions explicit to reduce layout shift
- compress before upload

## Licensing workflow

Before using any non-original image, record:

- source page URL
- creator name if required
- license type
- attribution text required, if any
- whether modification is allowed

For Wikimedia Commons and Openverse-sourced assets, verify the license on the original file page rather than trusting search metadata alone.

## Safety and trust

Do not present mockups, concept art, or edited composites as if they were real product or event photography.

If the article makes factual claims about a product, rely on real screenshots, official imagery, or source-backed visuals for the body content.

## Default article image package

Unless the user asks for something else, prepare:

- 1 hero image
- 1 to 3 in-body visuals for substantial how-to or comparison posts

For short posts, a single strong hero image may be enough.

The hero image should be attached during Ghost draft creation, not left for a separate manual content pass.

## Source preferences

Use these source classes in this order:

1. our own screenshots and photos
2. official press rooms and media kits with clear editorial use
3. Wikimedia Commons
4. Openverse with manual verification
5. Unsplash or Pexels for generic context only
